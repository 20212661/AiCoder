## 1. 架构层面改进

### 1.1 🔴 项目命名不一致（高优先级）

**问题**：项目存在三个不同名字，令人困惑：
- 仓库名：`AiCoder`
- pyproject.toml 中的包名：`rune`
- 实际包目录：`aicoder/`
- CLI 入口点：`rune = aicoder.main:main`

**建议**：统一命名，建议全部改为 `aicoder`：
```toml
# pyproject.toml
[project]
name = "aicoder"
# ...
[project.scripts]
aicoder = "aicoder.main:main"
```

### 1.2 🔴 强制 TUI 模式，CLI 代码路径成为死代码（高优先级）

**问题**：`main.py` 第202行硬编码 `args.tui = True`，导致 CLI 模式的代码（第336-365行）永远不会执行。`--no-stream` 等CLI相关参数形同虚设。

**建议**：
- 移除 `args.tui = True` 硬编码，恢复用户选择权
- 或者彻底移除 CLI 死代码，只保留 TUI 模式
- 如果保留双模式，添加集成测试覆盖两条路径

### 1.3 🟡 模块间依赖关系不清晰（中优先级）

**问题**：
- 延迟导入不一致：部分模块（session, repo, approval）使用延迟导入，部分直接导入
- `main.py` 中的步骤编号混乱（0→1→2→3→4→5→6→5→5.5→5.6→6→7）
- Coder 创建后才初始化 Git 仓库（步骤5.5/5.6），存在初始化顺序依赖问题

**建议**：
- 引入依赖注入容器或显式初始化流程
- 重新编排 `main()` 流程，使用有语义的标签代替数字编号
- 将 Git 初始化纳入 Coder 创建流程

### 1.4 🟡 缺少插件/扩展机制（中优先级）

**问题**：工具和命令都是硬编码注册的，没有提供用户自定义扩展的能力。

**建议**：
- 设计插件系统，允许用户自定义工具和命令
- 使用注册表模式 + 装饰器自动发现
- 参考 Aider 的 `--custom-tool` 机制

## 2. 核心模块改进

### 2.1 `main.py` — 主编排器

| 问题 | 严重性 | 建议 |
|------|--------|------|
| `_check_api_key` 匹配过于宽松（子串匹配） | 🔴 高 | 改用精确前缀匹配或配置映射表 |

| Git 初始化顺序问题 | 🟡 中 | 移入 Coder 工厂方法 |

### 2.2 `llm.py` — LLM 延迟加载

| 问题 | 严重性 | 建议 |
|------|--------|------|
| `_lazy_module` 无线程安全保护 | 🟡 中 | 添加 `threading.Lock` |
| `os.environ` 设置在多处重复 | 🟢 低 | 统一到一处设置 |
| 仅38行，功能单一但缺少错误处理 | 🟡 中 | 添加 import 失败的友好错误提示 |

**改进示例**：
```python
import threading

class LazyLiteLLM:
    def __init__(self):
        self._lazy_module = None
        self._lock = threading.Lock()

    def _load_litellm(self):
        if self._lazy_module is None:
            with self._lock:
                if self._lazy_module is None:  # double-check
                    import litellm
                    litellm.suppress_debug_info = True
                    self._lazy_module = litellm
        return self._lazy_module
```

### 2.3 `models.py` — 模型配置

| 问题 | 严重性 | 建议 |
|------|--------|------|
| 模型列表硬编码，更新需改代码 | 🟡 中 | 支持从远程/本地 JSON 加载模型列表 |
| 缺少模型能力声明 | 🟡 中 | 添加 `supports_vision`, `supports_tools` 等元数据 |
| 没有模型验证/测试 | 🟡 中 | 添加每个模型的连通性测试 |

### 2.4 `tui_app.py` — TUI 终端界面

| 问题 | 严重性 | 建议 |
|------|--------|------|
| 60fps 刷新率过高，浪费CPU | 🟡 中 | 改为事件驱动或降低到30fps |
| 缺少键盘快捷键系统 | 🟡 中 | 添加可自定义的快捷键绑定 |
| 错误处理不够友好 | 🟡 中 | 添加全局错误边界和友好提示 |

## 4. 代码质量问题

### 4.1 类型标注不完整

**现状**：项目配置了 mypy，但大量函数缺少返回类型和参数类型标注。

**建议**：
```python
# Before
def format_messages(self, user_msg):
    ...

# After
def format_messages(self, user_msg: str) -> list[dict[str, Any]]:
    ...
```

启用严格的 mypy 配置：
```toml
[tool.mypy]
strict = true
```

### 4.2 错误处理不统一

**问题**：
- 有些模块用 `logging.error()` + 返回 None
- 有些直接 `raise Exception`
- 有些静默忽略错误
- 用户看到的错误信息格式不一致

**建议**：
- 定义项目级异常体系：`AiCoderError`, `ConfigError`, `LLMError`, `ToolError`
- 统一错误处理策略和日志格式
- 在 UI 层统一展示错误信息

### 4.3 硬编码魔法值

**示例**：
- `base_coder.py` 中 `for _ in range(5)` — 工具调用最大次数
- `history.py` 中 token 阈值硬编码
- 上下文窗口截断比例硬编码

**建议**：提取为命名常量或配置项：
```python
MAX_TOOL_CALL_ROUNDS = 5
CONTEXT_WINDOW_RESERVE_RATIO = 0.15
SUMMARY_TRIGGER_RATIO = 0.85
```

### 4.4 日志系统不规范

**问题**：混合使用 `print()` 和 `logging`，缺少结构化日志。

**建议**：
- 全面采用 `logging` 模块
- 添加结构化日志格式（JSON 可选）
- 支持日志级别动态调整（如 `/loglevel debug` 命令）

## 5. 配置与工程化改进

### 5.1 配置管理

**问题**：
- `~/.aicoder/settings.json` 缺少 schema 验证
- 环境变量与配置文件的关系不清晰
- 没有配置文件模板或初始化命令

**建议**：
- 使用 pydantic 定义配置 schema
- 添加 `aicoder config init` 命令生成配置模板
- 实现配置优先级链：CLI参数 > 环境变量 > 配置文件 > 默认值

### 5.2 依赖管理

**问题**：
- `requirements.txt` 与 `pyproject.toml` 重复定义依赖
- 没有版本锁定（无 `requirements.lock` 或 `poetry.lock`）
- pyproject.toml 中 homepage URL 是占位符

**建议**：
- 移除 `requirements.txt`，只保留 `pyproject.toml`
- 添加版本锁定文件
- 修正项目元数据 URL

### 5.3 代码格式化和 Lint

**问题**：
- ruff 只配置了 line-length，没有启用任何 lint 规则
- 没有 pre-commit hooks
- 代码风格不完全一致

**建议**：
```toml
# pyproject.toml
[tool.ruff]
line-length = 100
select = ["E", "F", "I", "W", "N", "UP", "B", "A", "SIM"]

[tool.ruff.format]
quote-style = "double"
```

添加 `.pre-commit-config.yaml`：
```yaml
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.4.0
    hooks:
      - id: ruff
      - id: ruff-format
  - repo: https://github.com/pre-commit/mirrors-mypy
    rev: v1.10.0
    hooks:
      - id: mypy
```

## 7. 🎯 UI 系统重构：从 Web UI 到 TypeScript 桌面端 TUI

### 7.1 现状分析

**当前问题**：
- `aicoder-ui/` 使用 React + Vite Web 技术栈，需要浏览器运行，偏离 CLI 工具定位
- `aicoder/tui_app.py` 使用 Python Textual 框架，受限于 Python 生态的终端渲染能力
- 两套 UI 系统并存，维护成本高，功能重复

**重构目标**：用 **TypeScript** 编写原生桌面端 TUI，替代 Web UI 和 Python TUI，实现：
- ✅ 真正的终端原生体验，无需浏览器
- ✅ 高性能渲染（Ink 框架基于 Yoga 布局引擎）
- ✅ TypeScript 类型安全 + 丰富的 npm 生态
- ✅ 跨平台（Windows/macOS/Linux 终端）
- ✅ 与 Python 后端通过 stdio JSON-RPC 通信

### 7.2 推荐技术栈
 
| 层级 | 技术选型 | 理由 |
|------|---------|------|
| **TUI 框架** | [Ink](https://github.com/vadimdemedes/ink) (React for CLI) | React 模型开发 TUI，Yoga 布局引擎，成熟生态 |
| **语言** | TypeScript 5.x | 类型安全，IDE 支持好 |
| **构建** | tsup / esbuild | 快速打包为单文件 CLI |
| **Markdown 渲染** | marked + terminal-kit / chalk | 终端富文本渲染 |
| **语法高亮** | cli-highlight (基于 highlight.js) | 终端代码着色 |
| **状态管理** | zustand (轻量) | Ink 社区常用，API 简洁 |
| **后端通信** | JSON-RPC over stdio (子进程) | 无需 WebSocket，通过 stdin/stdout 通信 |
| **包管理** | pnpm | 快速、节省磁盘空间 |
| **发布** | nexe / pkg 打包为单二进制 | 用户无需安装 Node.js |

#### 7.5.2 RPC 通信层 `src/rpc/client.ts`

```typescript
import { spawn, ChildProcess } from 'child_process';
import { EventEmitter } from 'events';

interface JsonRpcRequest {
  jsonrpc: '2.0';
  id: number;
  method: string;
  params?: unknown;
}

interface JsonRpcNotification {
  jsonrpc: '2.0';
  method: string;
  params?: unknown;
}

export class RpcClient extends EventEmitter {
  private proc: ChildProcess | null = null;
  private idCounter = 0;
  private pending = new Map<number, { resolve, reject, timer }>();
  private buffer = '';

  async connect(pythonPath = 'python', args = ['-m', 'aicoder', '--serve']) {
    this.proc = spawn(pythonPath, args, {
      stdio: ['pipe', 'pipe', 'pipe'],
    });

    this.proc.stdout!.on('data', (data) => {
      this.buffer += data.toString();
      this.processBuffer();
    });

    this.proc.stderr!.on('data', (data) => {
      this.emit('stderr', data.toString());
    });

    this.proc.on('close', (code) => {
      this.emit('close', code);
    });
  }

  async request(method: string, params?: unknown, timeout = 30000): Promise<unknown> {
    const id = ++this.idCounter;
    const msg: JsonRpcRequest = { jsonrpc: '2.0', id, method, params };

    return new Promise((resolve, reject) => {
      const timer = setTimeout(() => {
        this.pending.delete(id);
        reject(new Error(`RPC timeout: ${method}`));
      }, timeout);

      this.pending.set(id, { resolve, reject, timer });
      this.proc!.stdin!.write(JSON.stringify(msg) + '\n');
    });
  }

  notify(method: string, params?: unknown) {
    const msg: JsonRpcNotification = { jsonrpc: '2.0', method, params };
    this.proc!.stdin!.write(JSON.stringify(msg) + '\n');
  }

  private processBuffer() {
    const lines = this.buffer.split('\n');
    this.buffer = lines.pop()!; // 保留未完成的行

    for (const line of lines) {
      if (!line.trim()) continue;
      try {
        const msg = JSON.parse(line);
        if (msg.id && this.pending.has(msg.id)) {
          const { resolve, timer } = this.pending.get(msg.id)!;
          clearTimeout(timer);
          this.pending.delete(msg.id);
          resolve(msg.result);
        } else if (msg.method) {
          // 后端推送的通知
          this.emit(msg.method, msg.params);
        }
      } catch (e) {
        this.emit('parse-error', line);
      }
    }
  }

  async disconnect() {
    this.proc?.kill();
    this.proc = null;
  }
}
```

```typescript
import chalk from 'chalk';
import { highlight } from 'cli-highlight';

export function renderMarkdownToTerminal(md: string): string {
  let result = md;

  // 代码块 ```lang ... ```
  result = result.replace(/```(\w+)?\n([\s\S]*?)```/g, (_, lang, code) => {
    return highlight(code.trim(), { language: lang || 'plaintext' });
  });

  // 行内代码
  result = result.replace(/`([^`]+)`/g, (_, code) => chalk.cyan(code));

  // 粗体
  result = result.replace(/\*\*([^*]+)\*\*/g, (_, text) => chalk.bold(text));

  // 斜体
  result = result.replace(/\*([^*]+)\*/g, (_, text) => chalk.italic(text));

  // 标题
  result = result.replace(/^### (.+)$/gm, (_, text) => chalk.bold.yellow(text));
  result = result.replace(/^## (.+)$/gm, (_, text) => chalk.bold.yellow.underline(text));
  result = result.replace(/^# (.+)$/gm, (_, text) => chalk.bold.yellow.underline.bgHex('#333')(text));

  // 列表
  result = result.replace(/^- (.+)$/gm, (_, text) => `  • ${text}`);
  result = result.replace(/^\d+\. (.+)$/gm, (_, text) => `  ◦ ${text}`);

  return result;
}
```

### 7.6 后端适配（Python 侧改造）

当前 `aicoder/io.py` 中的 `InputOutput` 基类已经定义了良好的接口。需要新增一个 `JsonRpcIO` 子类：

```python
# aicoder/rpc_io.py (新增文件)
class JsonRpcIO(InputOutput):
    """通过 JSON-RPC stdio 与 TypeScript TUI 通信的 IO 实现"""

    def __init__(self):
        self.reader = sys.stdin
        self.writer = sys.stdout
        self.pending_approvals = {}  # id -> Future

    def tool_output(self, message, bold=False):
        self._notify("tool_output", {"message": message, "bold": bold})

    def print_streaming(self, chunk):
        self._notify("streaming_chunk", {"text": chunk})

    def confirm_ask(self, question, default="y"):
        approval_id = str(uuid.uuid4())
        self._notify("approval_request", {"id": approval_id, "question": question})
        # 阻塞等待 TUI 回复
        return self._wait_response(approval_id)

    def get_input(self, root, rel_fnames, addable_rel_fnames, commands, ...):
        self._notify("input_request", {"root": root, ...})
        return self._wait_response("input")

    def _notify(self, method, params):
        msg = json.dumps({"jsonrpc": "2.0", "method": method, "params": params})
        self.writer.write(msg + "\n")
        self.writer.flush()

    def _wait_response(self, request_id):
        for line in self.reader:
            msg = json.loads(line)
            if msg.get("id") == request_id:
                return msg.get("result")
```
