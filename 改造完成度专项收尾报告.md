更新时间：2026-05-08

## 结论摘要

本次核对覆盖两份实施方案：

1. `Ink改造TUI实施方案.md`
2. `LangGraph重构后端实施方案.md`

整体结论：

- `official-ink` TUI 主路径已经落地，属于“主骨架已完成，工程收口未完成”。

建议判断：

- TUI 改造完成度：约 `75%`

## 三、逐项缺口清单

### P0：必须优先解决

1. 前端 `typecheck` 全局失败。
说明：当前新 TUI 虽已默认启用，但工程层没有稳定类型护栏，后续每次改动都难以判断是否引入回归。

2. `phase` 没有形成后端到前端的完整展示闭环。
说明：两份方案都把 phase 作为关键状态，但当前 UI 实际只稳定展示 `mode/model/streaming/approval`。

3. 模式/模型/命令切换需要做一次完整人工回归。
说明：虽然本次已修掉“切换模式后回答没变化”的关键根因，但仍需要把 `/plan`、`/act`、`/model`、`/yolo`、审批、取消生成整套走通。

### P1：建议下一阶段完成

1. 清理或隔离旧 `src/ink/` 高噪音存量。
说明：现在 legacy fallback 合理，但旧实现规模过大，已经明显影响类型收口与维护判断。

2. 让 LangGraph 审批统一走 `interrupts.py` 抽象。
说明：现在 interrupt 模块存在，但主 workflow 没有真正用它形成统一路径。

3. checkpoint 端到端闭环。
说明：需要真正验证“发起审批 -> 退出进程 -> 重启恢复 -> 继续审批”的完整链路。

4. 把 `status/update` 扩展到 `phase/checkpointId` 并在 TUI 展示。

### P2：可以后置

1. 扩展更多模式：`default / acceptEdits / bypassPermissions`
2. plan approval 专门 UI
3. tool result 可展开视图
4. token/cost/checkpoint/status 更丰富状态栏
5. 更彻底的旧 runtime 职责下沉与模块拆分
