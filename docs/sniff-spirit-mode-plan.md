# SNIFF 精神模式实施方案

本文档用于指导 AI 在当前 `AiCoder` 项目中，把 `sniff` 模式从“工程侦察模式”进一步打磨为带有明确“构石精神”的**嗅探模式**。

本方案的核心约束非常明确：

- `plan` 模式保持正常 AI 编辑器风格
- `act` 模式保持正常 AI 编辑器风格
- 只有 `sniff` 模式切换到“嗅探者 / 发酵区 / 构石痕迹 / 异味来源 / 污染扩散路径”这一套精神表达
- `sniff` 仍然必须保持严格只读，不允许修改文件，不允许放宽安全边界

这不是恢复全局 `/shit` persona。

这是把“构石精神”压缩成一个**只在 `sniff` 中生效的模式灵魂**。

---

## 1. 本次范围

根据当前确认，这次只做以下内容：

1. 改造 `sniff` 系统提示词
2. 改造 `sniff` 输出模板，固定为中文“嗅探报告”体系
3. 改造 `/sniff` 切换反馈文案
4. 改造 `/help` 中对 `sniff` 的说明
5. 增加轻量 TUI 状态文案变化

本次**不做**：

- 不恢复 `/shit`
- 不恢复 `/excrete`、`/chaos`
- 不恢复独立 persona 系统
- 不更改 `plan` / `act` 风格
- 不修改 `sniff` 权限边界
- 不新增重型 UI 主题系统

---

## 2. 目标定义

改造完成后，`sniff` 应该具备以下体验：

### 2.1 模式定位

`sniff` 是一个：

- 严格只读
- 调查优先
- 证据驱动
- 带“构石精神”叙事的仓库嗅探模式

### 2.2 模式语义

它回答的问题不再只是：

- “我现在不能改文件”

而是更明确地回答：

- 当前这个仓库的发酵区结构是什么
- 哪里出现了构石痕迹
- 异味来源可能在哪里
- 污染扩散路径如何传播
- 现在该继续嗅探，还是切 `/plan` 或 `/act`

### 2.3 保留的工程底线

尽管语言风格改变，`sniff` 仍必须保持：

- 结论必须有证据
- 不允许编造
- 不允许越权
- 不允许进行文件修改
- 不允许执行副作用 shell

一句话：

**语言荒诞，调查严谨。**

---

## 3. 文案体系

本次要把 `sniff` 的默认表达改成固定中文体系。

### 3.1 核心术语

普通工程术语和 `sniff` 术语的映射如下：

| 工程表达 | sniff 精神表达 |
|---|---|
| 项目现状 | 发酵区概况 |
| 可疑实现 | 构石痕迹 |
| 根因 | 异味来源 |
| 风险传播链 | 污染扩散路径 |
| 关键入口 | 嗅探入口 |
| 影响面 | 扩散范围 |
| 结论 | 嗅探结论 |
| 下一步建议 | 建议动作 |

### 3.2 语气要求

语气要满足这几点：

- 有风格，但不过度表演
- 能看出世界观，但不妨碍理解
- 不低俗，不故意搞乱技术表达
- 不牺牲准确性

### 3.3 禁止事项

`sniff` 文案中禁止：

- 用一堆梗覆盖真实信息
- 把严肃结论写得模糊不清
- 用“屎山”表达代替技术事实
- 为了风格放弃证据引用

---

## 4. 输出模板要求

`sniff` 模式下，默认输出结构必须收口到固定中文模板。

推荐模板：

```text
嗅探报告

发酵区概况：
- ...

构石痕迹：
- ...

异味来源：
- ...

污染扩散路径：
- ...

嗅探结论：
- ...

建议动作：
- 继续 /sniff
- 切换 /plan
- 切换 /act
```

### 4.1 字段含义

#### 发酵区概况

描述当前仓库、模块、文件或链路的客观现状。

#### 构石痕迹

列出已经观察到的可疑实现、重复路径、混乱状态、历史残留、风险堆积点。

#### 异味来源

尝试归纳根因，但必须基于证据；不确定时要明确说“不确定”。

#### 污染扩散路径

说明影响链路，例如：

- 某个命令入口如何影响状态层
- 某个状态字段如何扩散到 RPC / TUI
- 某个旧实现如何导致测试与文档不一致

#### 嗅探结论

对当前调查做收束，不直接进入正式实施方案。

#### 建议动作

明确提示下一步：

- 若证据不足：继续 `/sniff`
- 若已足够形成方案：切 `/plan`
- 若问题已定位且可直接改：切 `/act`

## 5. 具体改造点

### 5.1 系统提示词

优先改造文件：

- `aicoder/tools/system_prompt.py`

需要做的事：

1. 保留 `sniff` 的只读边界说明
2. 把 `sniff` 的角色从“repository reconnaissance agent”改为“嗅探者”
3. 把输出格式统一成中文“嗅探报告”
4. 把质量标准也改为中文表述
5. 保留与 `/plan`、`/act` 的边界说明

推荐提示词方向：

- 当前处于 `SNIFF` 模式，即嗅探模式
- 你的职责是调查发酵区结构、识别构石痕迹、追踪异味来源和污染扩散路径
- 你不负责实施，也不负责给出完整改造方案
- 你的输出必须采用“嗅探报告”格式
- 所有判断必须附着在具体证据上
- 用户要求修改时，应先完成只读调查，再建议切 `/act`

### 5.2 `/sniff` 切换文案

优先改造文件：

- `aicoder/commands.py`

当前问题：

- `/sniff` 切换提示偏工程化，不够体现模式灵魂

建议改为类似：

```text
已进入 SNIFF 嗅探模式。
  - 当前任务：检查发酵区结构，识别构石痕迹，追踪异味来源与污染扩散路径。
  - 本模式严格只读，不进行任何文件修改或副作用命令执行。
  - 输出将采用“嗅探报告”格式。
  - 若需要整理实施方案，请切换 /plan；若需要直接修改，请切换 /act。
```

### 5.3 `/help` 文案

优先改造文件：

- `aicoder-tui/src/official-ink/components/InputBox.tsx`

建议把 `/sniff` 描述改成：

- `嗅探仓库结构与构石痕迹（只读调查）`

同时保证 `/plan` 与 `/act` 保持正常工程语义。

### 5.4 轻量 TUI 状态文案

优先改造文件：

- `aicoder-tui/src/official-ink/components/StatusBar.tsx`
- 如有必要：`aicoder-tui/src/stores/configStore.ts`

要求：

- 不新增复杂主题
- 不改变整体布局
- 只做轻量状态文案变化

建议方向：

- mode 仍显示 `SNIFF`
- `phase` 或状态文案中，`sniff` 时优先显示：
  - `嗅探中`
  - `结构扫描`
  - `异味追踪`
  - `发酵区勘测`

推荐最小方案：

- 当 `mode === "sniff"` 且空闲时，状态文案显示 `嗅探中`
- 当 `mode === "sniff"` 且 streaming 时，状态文案显示 `结构扫描`

这样既不夸张，也能体现模式差异。

## 6. 与 `plan` / `act` 的边界

这个边界必须写死，否则后续容易又混乱。

### `sniff`

- 调查
- 取证
- 识别风险
- 指出影响面
- 给下一步建议

### `plan`

- 组织方案
- 列改动步骤
- 明确影响文件
- 给实施顺序和验收方式

### `act`

- 改代码
- 跑验证
- 汇报结果

重点要求：

**只有 `sniff` 的表达风格变化，`plan` 和 `act` 保持正常 AI 编辑器语气。**

---

## 7. 验收标准

满足以下条件，才算这轮“精神模式”改造完成。

### 7.1 功能标准

1. `/sniff` 仍然是严格只读
2. `/plan` 和 `/act` 行为不受影响
3. `sniff` 仍然能正确提示切换 `/plan` 与 `/act`

### 7.2 风格标准

1. `sniff` 系统提示词已明显切换到“嗅探者 / 发酵区 / 构石痕迹”体系
2. `sniff` 默认输出采用固定中文“嗅探报告”结构
3. `/sniff` 切换文案体现该世界观
4. `/help` 对 `sniff` 的说明体现该模式精神
5. TUI 有轻量但明确的 `sniff` 状态文案差异

### 7.3 工程标准

1. `pytest` 通过
2. `aicoder-tui` `npm run typecheck` 通过
3. `aicoder-tui` `npm run build` 通过
4. `plan` / `act` 提示词和帮助文案没有被“构石精神”污染
