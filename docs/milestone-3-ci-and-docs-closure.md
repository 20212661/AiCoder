# 里程碑 3：CI 和文档闭环

## 目标

建立最小可依赖的质量门，并让项目文档和实际代码状态保持一致，形成稳定的开发闭环。

本里程碑完成后，应达到以下状态：

- 后端测试有基础 CI 守护
- TUI typecheck 有基础 CI 守护
- README 与当前实现一致
- 核心设计文档、权限文档、协议文档齐备
- 新加入的开发者可以根据文档启动和理解项目

## 背景问题

当前项目虽然有较多后端测试，但仍存在这些问题：

- TUI 失败不会被统一质量门拦住
- README 与实际实现脱节
- 协议、权限、运行时设计没有形成闭环文档
- 改完代码后，外部协作者很难快速理解当前状态

## 本里程碑范围

要做：

- 建最小 CI
- 修 README
- 整理里程碑内产出的关键文档
- 输出阶段总结和下一阶段 backlog

不做：

- 复杂发布流水线
- 全量性能测试体系
- 过重的 DevOps 重构

## AI 执行总要求

请按以下原则执行：

1. 先保证最小有效，再考虑扩展。
2. CI 命令必须与本地开发命令一致。
3. 文档必须描述当前真实状态，不夸大完成度。
4. 如果某项仍未完全通过，必须明确写为已知问题。

---

## 任务 1：修复 README 与启动说明

### 目标

让 README 与当前项目真实结构、启动方式、能力边界一致。

### 需要阅读

- `README.md`
- `pyproject.toml`
- `aicoder/main.py`
- `aicoder/__main__.py`
- `aicoder-tui/package.json`

### 需要执行

1. 找出 README 中所有过时内容。
2. 重写或修正：
   - 项目简介
   - 当前架构概览
   - Python CLI 启动方式
   - `--serve` 模式说明
   - TUI 启动方式
   - plan / act 模式说明
   - 权限与审批说明
   - 测试命令
   - 已知限制

### 验收标准

- 新同学只看 README 就能正确启动项目
- README 不再引用旧结构或旧命令

---

## 任务 2：整理核心设计文档

### 目标

把前两个里程碑的关键设计结论变成稳定文档资产。

### 需要执行

确认并整理这些文档是否存在、是否与代码一致：

- `aicoder/docs/runtime-unification.md`
- `aicoder/docs/permission-matrix.md`
- `aicoder/docs/rpc-protocol.md`
- `aicoder-tui/docs/typecheck-triage.md`

必要时补充：

- 当前状态说明
- 设计取舍
- 已知限制
- 后续 backlog 提示

### 验收标准

- 关键设计文档完整可读
- 文档和当前实现没有明显冲突

---

## 任务 3：加最小 CI 质量门

### 目标

避免再次出现“后端测试通过但 TUI 已损坏”的情况。

### 需要阅读

- `pyproject.toml`
- `aicoder-tui/package.json`
- 仓库现有 CI 配置（如果有）

### 需要执行

1. 新增或修改 CI 配置。
2. 最小 CI 至少包含：
   - Python：`pytest`
   - TUI：`npm run typecheck`
3. 如果 TUI 已能 build，可追加：
   - `npm run build`
4. 保持 CI 轻量，不引入过重流程。

### 验收标准

- 主分支具备最小回归防护
- CI 命令和本地开发命令一致

---

## 任务 4：补最小冒烟验证

### 目标

除了单测和 typecheck，再补最基础的运行层验证。

### 需要执行

按当前项目实际情况，尽量补最小冒烟验证，例如：

- CLI 能启动
- `--serve` 模式能启动
- 若 TUI 可运行，记录最小联调方式

如果不适合做自动化，也至少把手工验证步骤写入文档。

### 验收标准

- 至少有一套明确的运行验证方法
- 不是只有测试，没有真实启动验证

---

## 任务 5：输出本轮阶段总结

### 目标

给后续 AI 或开发者一个干净的接手点。

### 需要执行

新建总结文档：

- `docs/iteration-1-summary.md`

文档至少包含：

- 本轮完成项
- 本轮未完成项
- 当前已知风险
- 当前推荐开发命令
- 下一阶段 backlog

推荐 backlog 方向：

- repo map / 索引增强
- 上下文管理增强
- 更稳定的 TUI 架构收口
- 更强的 E2E 测试

### 验收标准

- 后续接手者能快速知道当前项目进度
- 不是只看 commit 猜项目状态

---

## 建议执行顺序

1. 修复 README 与启动说明
2. 整理核心设计文档
3. 加最小 CI 质量门
4. 补最小冒烟验证
5. 输出本轮阶段总结

---

## 每次执行后的输出格式

请严格按以下格式汇报：

1. 分析
   - 看了哪些文档和配置
   - 当前闭环缺口在哪里
2. 实施
   - 改了哪些文件
   - 补了哪些文档或 CI
3. 验证
   - 跑了哪些命令
   - CI/测试/typecheck 状态如何
4. 总结
   - 当前闭环到了什么程度
   - 还剩哪些风险
   - 下一步建议是什么
