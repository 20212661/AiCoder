# PRD: SNIFF 结构化侦察增强

## 1. 概述

当前 `sniff` 模式已经具备：

- 独立模式语义
- 严格只读权限边界
- 中文“嗅探报告”提示词
- 运行时附加消息约束
- 轻量 TUI 文案差异

但它目前的结构化输出仍然**主要依赖模型临场发挥**。虽然提示词已经要求输出“发酵区概况 / 构石痕迹 / 异味来源 / 污染扩散路径 / 嗅探结论 / 建议动作”，但这些内容还没有充分得到后端运行时的程序化支持。

本次功能的目标是：

**让 `sniff` 在保持严格只读的前提下，具备更稳定的程序化侦察摘要生成能力，通过后端辅助生成侦察骨架、仓库概况和调查提示，从而减少模型输出完全依赖临场发挥的程度。**

这次不是要让后端“替模型完成分析”，而是要让系统提供一层高质量的侦察支架，帮助模型更稳定地生成专业且风格统一的“嗅探报告”。

---

## 2. 目标

- 让 `sniff` 的输出结构更稳定，不因模型波动而大幅漂移
- 让首次嗅探时自动附带基础仓库侦察摘要
- 让模型更容易识别：
  - 发酵区概况
  - 嗅探入口
  - 构石痕迹
  - 可能的扩散范围
- 保持 `plan` 和 `act` 的现有正常风格，不被污染
- 保持 `sniff` 严格只读

---

## 3. 用户故事

### US-001: 为 `sniff` 生成程序化侦察骨架

**描述：** 作为用户，我希望 `sniff` 在进入调查时自动获得一份结构化侦察骨架，这样输出不会完全依赖模型临场组织。

**验收标准：**
- [ ] 新增独立的 `sniff` 辅助模块用于生成侦察骨架
- [ ] 骨架至少包含：发酵区概况、嗅探入口、构石痕迹候选、扩散范围候选、建议动作
- [ ] 该骨架只在 `sniff` 模式下注入
- [ ] 不引入写文件或副作用命令
- [ ] `pytest` 通过

### US-002: 为首次仓库嗅探提供自动摘要

**描述：** 作为用户，我希望 `sniff` 在第一次调查陌生仓库时自动拿到一份仓库结构摘要，这样模型能更快进入有效调查。

**验收标准：**
- [ ] 自动摘要至少包含根目录关键结构、测试目录、配置入口、主要代码入口候选
- [ ] 摘要内容来自现有只读能力，不新增危险操作
- [ ] 若仓库信息不足，能优雅降级为空摘要
- [ ] 不影响 `plan` / `act` 的消息构建
- [ ] `pytest` 通过

### US-003: 让 `sniff` 输出更稳定地贴合“嗅探报告”结构

**描述：** 作为用户，我希望 `sniff` 的回答默认更稳定地落在“嗅探报告”六段结构上，而不是偶尔退回普通分析风格。

**验收标准：**
- [ ] `system_prompt` 中继续约束中文“嗅探报告”结构
- [ ] `message attachment` 中引用程序化侦察骨架
- [ ] `sniff` 模式消息里明确包含结构化字段提示
- [ ] `plan` / `act` 不包含这些 `sniff` 专属结构字段
- [ ] 对应测试覆盖通过

### US-004: 让 `sniff` 更擅长做首次仓库侦察

**描述：** 作为用户，我希望 `sniff` 在我首次进入一个项目时能更像侦察员，而不是普通只读助手。

**验收标准：**
- [ ] `sniff` 模式优先附带“陌生仓库侦察顺序”提示
- [ ] 自动摘要与提示词语义一致
- [ ] 嗅探模式下能优先看到结构概况而不是空泛开场
- [ ] 不修改现有权限模型
- [ ] `pytest` 通过

### US-005: 保持模式边界清晰

**描述：** 作为维护者，我希望 `sniff` 的增强不会污染 `plan` 和 `act`，避免再次造成模式语义混乱。

**验收标准：**
- [ ] 程序化侦察摘要只在 `sniff` 模式启用
- [ ] `plan` 和 `act` 的系统提示词保持现有正常风格
- [ ] `plan` / `act` 的 message attachment 不注入嗅探骨架
- [ ] 新增测试明确断言模式隔离
- [ ] `aicoder-tui` `typecheck` 和 `build` 通过

## 4. 功能需求

### FR-1: 新增独立 `sniff` 辅助模块

系统必须新增一个只服务于 `sniff` 的独立辅助模块，例如：

- `aicoder/sniffing/`
- 或 `aicoder/recon/`

推荐最小文件组织：

- `aicoder/sniffing/__init__.py`
- `aicoder/sniffing/recon_summary.py`

### FR-2: 程序化生成仓库侦察摘要

系统必须能在 `sniff` 模式下程序化生成一段可注入到消息中的仓库侦察摘要。

摘要建议包含：

- 发酵区概况：
  - 根目录关键目录
  - 后端/前端/测试目录
  - 配置文件候选
- 嗅探入口：
  - 主入口文件候选
  - 命令入口候选
  - 状态或工作流入口候选
- 构石痕迹候选：
  - 重复入口
  - 历史残留字段
  - 文档/实现不一致
  - 重复模式分叉
- 扩散范围候选：
  - 直接相关模块
  - 间接相关模块

### FR-3: 程序化摘要只可使用现有只读信息

摘要生成必须只依赖：

- 已有的仓库文件树
- 代码定义提取
- 只读搜索结果
- 只读命令结果

不得：

- 写文件
- 修改状态
- 运行副作用命令

### FR-4: `sniff` 模式消息中注入侦察骨架

`sniff` 模式的 runtime attachment 必须包含程序化侦察摘要或骨架提示，使模型能围绕该结构完成回答。

### FR-5: `plan` / `act` 不得注入嗅探摘要

程序化侦察摘要严禁泄漏到：

- `plan`
- `act`

### FR-6: 固定中文字段体系

`sniff` 程序化骨架和提示词必须统一采用以下字段：

- 发酵区概况
- 嗅探入口
- 构石痕迹
- 异味来源
- 污染扩散路径
- 嗅探结论
- 建议动作

### FR-7: 提供可降级行为

当仓库信息不完整时：

- 摘要可部分缺失
- 模块应返回空段或占位，而不是抛异常中断

### FR-8: 增加测试覆盖

必须补充测试，验证：

- `sniff` 摘要存在
- `sniff` 摘要字段正确
- 非 `sniff` 模式不包含该摘要
- 模块在仓库信息不足时能优雅降级

---

## 5. 非目标

本次不做：

- 不新增 GUI 复杂可视化
- 不恢复 persona 系统
- 不新增新的命令模式
- 不让后端替模型直接输出最终结论
- 不做完整静态分析引擎
- 不引入数据库或索引服务

---

## 6. 设计考虑

### 6.1 设计原则

- 侦察摘要是“支架”，不是“替代回答”
- 模型仍负责最后的调查语言组织
- 运行时负责尽可能给模型高信号上下文
- 避免把 `sniff` 变成“写死模板拼接器”

### 6.2 推荐摘要内容风格

程序化摘要可以用类似这样的内部结构：

```text
SNIFF RECON SUMMARY:
- 发酵区概况:
  - 根目录关键结构: ...
  - 配置入口候选: ...
- 嗅探入口:
  - 命令入口候选: ...
  - 主执行入口候选: ...
- 构石痕迹候选:
  - ...
- 扩散范围候选:
  - ...
```

然后由 `message_builder` 把它作为 `sniff` 模式附加消息的一部分注入。

### 6.3 生成时机建议

建议仅在以下条件下生成：

- 当前 mode == `sniff`
- 当前轮准备发给模型前

可以复用：

- 现有文件树信息
- `Coder` 上下文
- 已缓存仓库信息

---

## 7. 技术考虑

### 7.1 推荐落点

建议新增独立模块：

- `aicoder/sniffing/recon_summary.py`

建议提供函数：

- `build_sniff_recon_summary(coder) -> str`

可选拆分：

- `summarize_repo_structure(coder) -> dict`
- `detect_entry_points(coder) -> dict`
- `detect_recon_signals(coder) -> dict`

### 7.2 推荐接入点

推荐接入：

- `aicoder/coders/message_builder.py`

在 `build_mode_messages(coder)` 的 `sniff` 分支里调用新模块，生成 summary 并注入。

### 7.3 推荐优先信息源

优先从以下已存在能力提取：

- `coder._file_tree`
- `coder.get_repo_map()`
- `coder.root`
- 已缓存的 chat files / tracked files
- 代码定义提取能力

不建议第一版直接引入复杂新依赖。

## 8. 成功指标

- `sniff` 输出结构更稳定
- 首次仓库嗅探时更少出现空泛开场
- `sniff` 与 `plan` 的差异更明显
- 模式行为不回归
- 工程验证保持全绿
