# 可执行修复文档

更新时间：2026-05-08

## 1. 目标

本修复文档用于把当前改造状态从：

- `架构已落地，产品化收尾未完成`

推进到：

- `新 TUI 默认可用`
- `模式/模型/审批/取消生成行为一致`
- `LangGraph 路径具备稳定验证基础`

本文档只安排“下一轮必须做的收尾修复”，不继续扩展新功能。

---

## 2. 修复总原则

1. 先修“状态一致性”，再修“类型收敛”，最后补“checkpoint/interrupt 闭环”。
2. 优先保证默认路径稳定，不急着删除 legacy。
3. 每完成一批任务，都必须有明确验收，不以“代码看起来对”作为完成标准。
4. 不同时大改 TUI、RPC、LangGraph 三层协议，避免回归面过大。

---

## 3. 修复范围

本轮修复只覆盖以下三块：

1. `official-ink` 默认 TUI 路径
2. `mode/model/phase/approval` 状态链路
3. LangGraph runtime 的审批与 checkpoint 收尾

不在本轮范围内的事项：

1. 删除 `src/ink/` 旧实现
2. 扩展更多权限模式
3. 增加全新 UI 能力
4. 默认切换到 LangGraph runtime

---

## 4. 执行顺序

严格按以下顺序执行：

1. P0-A：修复状态链一致性
2. P0-B：补人工回归闭环
3. P1-A：收敛新 TUI 路径相关 TypeScript 错误
4. P1-B：补 phase 展示
5. P1-C：统一 LangGraph 审批入口
6. P2-A：打通 checkpoint 恢复闭环

---

## 5. P0-A：修复状态链一致性

### 目标

确保以下问题不再出现：

1. 切换 `/plan` 后 UI 显示变了，但回答仍像 `act`
2. 切换 `/model` 后前后端显示不一致
3. `/yolo` 只在前端变更，没有真正通知后端
4. 状态栏、输入框、本地 store、后端 runtime 之间使用不同状态源

### 修改范围

#### 后端

- [aicoder/rpc_io.py](D:/CodingProject/aiCoder/aicoder/rpc_io.py:1)
- [aicoder/coders/message_builder.py](D:/CodingProject/aiCoder/aicoder/coders/message_builder.py:1)
- [aicoder/coders/base_coder.py](D:/CodingProject/aiCoder/aicoder/coders/base_coder.py:1)
- [aicoder/graph/nodes.py](D:/CodingProject/aiCoder/aicoder/graph/nodes.py:1)

#### 前端

- [aicoder-tui/src/stores/configStore.ts](D:/CodingProject/aiCoder/aicoder-tui/src/stores/configStore.ts:1)
- [aicoder-tui/src/official-ink/components/InputBox.tsx](D:/CodingProject/aiCoder/aicoder-tui/src/official-ink/components/InputBox.tsx:1)
- [aicoder-tui/src/official-ink/components/StatusBar.tsx](D:/CodingProject/aiCoder/aicoder-tui/src/official-ink/components/StatusBar.tsx:1)
- [aicoder-tui/src/official-ink/components/ModelPicker.tsx](D:/CodingProject/aiCoder/aicoder-tui/src/official-ink/components/ModelPicker.tsx:1)
- [aicoder-tui/src/rpc/methods.ts](D:/CodingProject/aiCoder/aicoder-tui/src/rpc/methods.ts:1)

### 必做项

1. 后端发送的 `status/update` 必须统一带上：
   - `model`
   - `mode`
   - `planMode`

2. 模型侧上下文必须显式注入当前运行时状态：
   - 当前模型名
   - 当前模式

3. 前端只允许以 `mode` 作为主状态源。
   - `planMode` 只能作为兼容字段
   - UI 显示时优先取 `mode`

4. `/plan` 和 `/act` 必须显式设值，禁止继续使用“toggle”思路。

5. `/model` 必须形成 RPC 闭环：
   - 前端可请求切换
   - 后端真正接收并走 `/model xxx`
   - 切换后状态栏更新

6. `/yolo` 必须真正发给后端。

### 验收标准

手工验证以下场景：

1. 启动后提问“现在是什么模型”，回答必须与状态栏一致。
2. 输入 `/plan` 后提问“现在是什么模式”，回答必须明确为 `plan`。
3. 输入 `/act` 后提问“现在是什么模式”，回答必须明确为 `act`。
4. 切换模型后，状态栏和回答里的模型名一致。
5. 输入 `/yolo` 后，再看后端输出或审批行为，必须发生真实变化。

## 6. P0-B：补人工回归闭环

### 目标

把“改了代码”变成“行为已确认”。

### 执行方式

使用默认 official-ink 路径手工回归。

### 回归命令

在项目根目录：

```powershell
cd D:\CodingProject\aiCoder\aicoder-tui
cmd /c npm run dev
```

确保后端服务路径正常可起。

### 回归用例

#### 用例 1：模式识别

输入：

```text
/plan
现在是什么模式
```

期望：

- 状态栏显示 `PLAN`
- 回答明确说明当前是 `plan`

#### 用例 2：Act 识别

输入：

```text
/act
现在是什么模式
```

期望：

- 状态栏显示 `ACT`
- 回答明确说明当前是 `act`

#### 用例 3：模型切换

输入：

```text
/model
```

操作：

- 在 picker 中选择另一个模型
- 再提问“现在是什么模型”

期望：

- 状态栏更新
- 回答与状态栏一致

#### 用例 4：Plan 模式只读

输入：

```text
/plan
创建 hello.txt
```

期望：

- 不执行写入
- 回答提示需要切到 `/act`

#### 用例 5：Act 模式可执行

输入：

```text
/act
读取 README.md
```

期望：

- 能调用读工具
- 工具卡片和结果显示正常

#### 用例 6：审批

输入：

```text
/act
修改 README.md
```

期望：

- 出现审批面板
- `y/n/esc` 都有效

#### 用例 7：取消生成

输入一个会长输出的问题，在流式输出中按：

```text
Esc
```

期望：

- 生成中断
- UI 不死锁

### 验收标准

7 个用例全部通过，才能进入下一阶段。

## 7. P1-A：收敛新 TUI 路径相关 TypeScript 错误

### 目标

当前 `aicoder-tui` 全局 `typecheck` 失败，但本轮不要求一次性修完全部历史错误。目标是：

- 明确区分“历史错误”和“新路径错误”
- 至少把 `official-ink` 相关新增路径收敛到无新增错误

### 当前现状

`cmd /c npm run typecheck` 仍有大量错误，主要集中在：

1. `src/ink/*`
2. `src/design-system/*`
3. 第三方声明缺失
4. 一些旧路径引用错误

### 执行策略

1. 先记录当前全量错误输出。
2. 把错误按目录分组。
3. 优先处理这些目录中直接影响官方新路径的项目：
   - `src/official-ink/*`
   - `src/stores/*`
   - `src/rpc/*`

4. 如有必要：
   - 补 `.d.ts`
   - 修错误导出
   - 限制 `tsconfig` 检查范围用于阶段性验收

### 建议动作

1. 生成一份基线错误文件：

```powershell
cd D:\CodingProject\aiCoder\aicoder-tui
cmd /c npm run typecheck > ..\tsc-baseline.txt 2>&1
```

2. 提取 official-ink 相关错误：

```powershell
Select-String -Path ..\tsc-baseline.txt -Pattern "official-ink|stores|rpc"
```

3. 优先清掉 official-ink 自身报错。

### 验收标准

至少满足其一：

1. `official-ink` 路径无类型错误
2. 能明确证明当前 `typecheck` 失败均为历史存量，与新路径无关

## 9. P1-C：统一 LangGraph 审批入口

### 目标

避免现在“有 `interrupts.py`，但主路径没真正统一走它”的半成品状态。

### 修改范围

- [aicoder/graph/interrupts.py](D:/CodingProject/aiCoder/aicoder/graph/interrupts.py:1)
- [aicoder/graph/nodes.py](D:/CodingProject/aiCoder/aicoder/graph/nodes.py:1)

### 必做项

1. `permission_node` 统一通过 `interrupts.py` 请求审批
2. 无 checkpointer 时走 blocking fallback
3. 有 checkpointer 时走 native interrupt

### 目标结构

不要在 `permission_node` 里直接拼审批逻辑，统一变为：

```python
from .interrupts import request_tool_approval
```

### 验收标准

1. 无 checkpoint 模式下审批照常弹出
2. 打开 checkpoint 模式后不会报错
3. 审批拒绝后 observation 仍能回写

## 12. 完成定义

只有同时满足以下条件，才能认为这一轮修复完成：

1. `/plan` `/act` `/model` `/yolo` 行为一致
2. 回答中的当前模型/模式与状态栏一致
3. 审批面板可正常批准/拒绝
4. `Esc` 取消生成可靠
5. `phase` 已显示
6. 新 TUI 路径的类型问题至少已被隔离或说明清楚
7. LangGraph 审批入口已统一

如果 checkpoint 也打通，则可额外视为“第二阶段收尾完成”。
